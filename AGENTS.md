# 프로젝트 목표
- UI/UX 감각을 기르기 위해 데스크탑 형태의 인터페이스를 처음부터 직접 구현하며 다양한 UI 구성요소와 상호작용 흐름을 실습한다.
- Git을 통한 기본적인 작업 흐름과 협업 방법을 배운다.

# 기술 스택
- 개발하면서 기술 스택이 변경된 경우 이 문서도 꼭 같이 고쳐야 한다.
- 언어: TypeScript
- 플래폼: 웹브라우저 (데스크탑 / 모바일)
- 클라이언트: PixiJS v8
- 아이콘: Font Awesome Free Brands SVG를 사용한다.
- 빌드: `vite build`로 프로덕션 번들을 생성하고, 개발 중에는 `vite` 개발 서버를 사용한다.
- 이미지 애셋: `free-tex-packer`로 이미지를 PixiJS 스프라이트시트 (atlas PNG + JSON)로 패킹한다.
- 서버: 로컬 개발, 정적 파일 서빙, 점수 기록 등의 API를 담당한다.
- 테스트: Vitest 기반 단위 테스트를 사용한다.

# 파일 구조 규칙
- `src`는 아래 구조를 기본으로 사용한다.
  - `src/component`: 재사용 가능한 컴포넌트를 둔다.
  - `src/assets`: 이미지, 폰트, 스프라이트시트 등 정적 애셋을 둔다.
  - `src/hooks`: 재사용 가능한 훅을 둔다.
  - `src/pages`: 화면 단위 코드와 HTML 엔트리 파일을 둔다.
  - `src/constants`: 공통 상수를 둔다.
  - `src/config`: 설정값과 설정 관련 코드를 둔다.
  - `src/styles`: CSS 파일과 스타일 관련 코드를 둔다.
  - `src/service`: API 및 외부 통신 로직을 둔다.
  - `src/utils`: 공통 함수와 패턴을 둔다.
  - `src/contexts`: 전역 컨텍스트와 공유 상태 코드를 둔다.
  - `src/App.ts`: 앱의 루트 조립 코드를 둔다.
  - `src/index.ts`: 앱의 공통 부트스트랩을 둔다.
  - `src/pages/index.html`: 기본 페이지 HTML 엔트리를 둔다.
  - `src/pages/index.ts`: HTML 엔트리와 앱 부트스트랩을 연결하는 페이지 진입점을 둔다.
- 새 파일을 만들 때는 먼저 위 구조 안에 배치할 수 있는지 확인하고, 특별한 이유가 없으면 `src` 루트에 임의의 새 폴더나 새 파일을 추가하지 않는다.
- 구조가 변경되면 `AGENTS.md`도 함께 수정한다.

# 코딩 규칙
- 모든 코드는 짧고, 명확하고, 읽기 쉽게 유지한다.
- 필요한 경우 주석은 Simple English로 작성한다.
- 문서는 명확하고 모호하지 않은 한국어로 작성한다.
- 새 소스 파일 (테스트 제외)을 만들 때는 파일 최상단에 아래 형식의 멀티라인 블록 주석을 추가한다.
  /*
   * This file is for ...
   * Can be multiple lines ...
   */
- 들여쓰기는 2칸을 사용하고, 줄 끝 공백을 남기지 않으며, 모든 텍스트 파일의 마지막 줄 끝에는 개행 문자 1개를 둔다.
- TypeScript/JavaScript 파일에서는 줄 끝 세미콜론을 사용하지 않는다.
- 이 규칙에서 별도로 지정하지 않으면 각 언어의 스타일 가이드를 따른다.
- 의도가 명확하도록 `let`보다 `const`를 우선 사용한다. 반복문 내부나 실제 변경이 필요한 경우 `let` 사용은 허용한다.
- 타입 오류를 피하려고 `| null` 또는 `| undefined`를 임의로 추가하지 않는다. 논리적으로 `null` / `undefined`가 될 수 없는 값이라면 타입에서 허용하면 안 된다. 실제 런타임 동작상 해당 값이 가능할 때만 nullable 타입으로 만든다. `any`도 엄격하게 금지된다.

# 테스트
- 모든 기능마다 단위 테스트를 작성한다.
- 코딩 후에는 AI 에이전트가 직접 테스트를 실행해 통과를 확인한다.
- AI 에이전트는 게임을 눈으로 확인할 수 없으므로, 게임의 현재 상태를 텍스트로 덤프할 수 있는 기능을 만든다. 해당 덤프를 기준으로 유저 액션과 게임 상태 변화를 검증하는 테스트를 작성한다.

# Git 규칙
- 커밋 메시지는 Simple English로 작성하고 이모지를 사용하지 않는다.
- AI 에이전트 관련 내용을 커밋 메시지에 쓰지 않는다.
- 커밋 메시지는 Conventional Commits를 사용한다 (https://www.conventionalcommits.org/en/v1.0.0/).
- 필요할 때 멀티라인 커밋 메시지는 표준 형식 (Conventional Commit 제목 1줄, 빈 줄, 본문)으로 작성한다.
- Fast-forward 머지는 사용하지 않는다. 기능별 브랜치를 사용하고, 머지 커밋으로 병합한다 (e.g. `git merge --no-ff`).
- Git Flow 스타일 브랜치 모델을 사용한다.
  - `master`: 항상 릴리스 가능한 상태를 유지하며, `release/...` (및 긴급 수정)에서만 머지한다.
  - `develop`: 기본 통합 브랜치이며, 일상 작업의 기준 브랜치로 사용한다.
  - `feature/<short-name>`: `develop`에서 분기해 새 기능 개발 후 `develop`으로 머지하고 삭제한다.
  - `fix/<short-name>`: `develop`에서 분기해 버그 수정 후 `develop`으로 머지하고 삭제한다.
  - `release/<version>`: `develop`에서 분기해 릴리스 안정화 후 `master`와 `develop` 모두에 머지한다. 릴리스 태그는 `master`에 생성한다 (e.g. `v0.1.0`).
  - 운영 긴급 수정은 `master`에서 `fix/<short-name>`를 분기하고, 완료 후 `master`와 `develop` 모두에 머지한다.
  - 위 규칙은 가이드라인이며 각 상황에 맞추어 유연하게 사용할 수 있다.
- 브랜치 이름은 소문자와 하이픈을 사용한다 (가능하면 이슈 ID 포함하되, 이슈 ID만으로 브랜치 이름을 생성하지 않고 간단한 설명과 함께 쓴다.).
- `master` / `develop` 병합은 PR 기반으로 진행하고, 직접 푸시는 막아 보호한다.

# AI 에이전트 규칙
- 사용자가 별도 언어를 지정하지 않으면, 컴퓨터에 설정된 언어로 응답한다.
- 사용자가 명시적으로 요청하지 않으면 Git 커밋을 만들지 않는다.
- 명령을 수행할 때 추측하지 말고, 혼란스러운 점은 명령자에게 다시 묻는다. 명령자가 의도적으로 자율성을 준 경우에는 자율적으로 진행할 수 있다.
- 명령 수행에 필요한 최소한의 코드만 작성하고, 필요 없는 과도한 추상화나 과도한 오류 처리를 하지 않는다.
  - 추후 추상화가 필요하거나, 추가로 발생할 수 있는 오류가 있다면 명령자에게 피드백한다.
  - 명령자는 필요한 경우 시스템 확장을 대비한 추상화 등 고수준 작업을 명령할 수 있다.
- 명확한 지시 없이 기능 작성/수정에 필요하지 않은 인접 코드, 주석, 서식을 개선하지 않는다.
  - 멀쩡하게 동작하는 코드를 AI 에이전트 취향으로 리팩터링하는 행위를 엄격히 금지한다.
  - 코드 작성/수정 중 파악한 개선점은 명령자에게 피드백한다.
  - 변경 사항으로 인해 쓰지 않는 함수나 파일이 생기면 삭제할 수 있다.
- 코드 작성 및 수정 후 모든 테스트 케이스를 통과해야 한다.
  - 명령자의 명확한 요청이 있으면 테스트를 스킵할 수 있다.
- 모든 규칙은 명령자의 명확한 명령이 있는 경우 `AGENTS.md`보다 명령을 우선한다. 단, `AGENTS.md`에 영속적으로 남아야 하는 내용은 명령자에게 피드백한다.
